# Known failure modes

Findings from the eval run on commit `f062417` (`gpt-4.1-mini`, 25 closed Bugs/Tasks
from Purview Data Governance, auto-drafted ground truth).

Source data: `evals/runs/2026-05-04T20-26-55Z/`.

> **Caveat — auto-labeled ground truth.**
> Labels were generated by the same model family (`gpt-4.1-mini`) via
> `evals/draft_labels.py`, so absolute scores are biased *toward* the model under
> test. The failure modes below are still informative because they show
> *systematic* drift — places where the production summarizer disagrees with even
> a same-family labeler that saw the same fields.

---

## 1. Risk hallucination (model invents 0.7 extra risks per item)

**Symptom.** The summarizer predicts **1.68 risks/item** but the labels expect
**1.00 risks/item**. Of the 42 predicted risks, only 12 align to an expected
risk; the remaining 30 are flagged as unsupported (1.20/item).

**Example** — work item 3921445 ("Actions still visible for deleted governance domains"):

> **Expected risks** (1):
> - "actions still visible / deleted governance domain" (Medium)
>
> **Predicted risks** (2):
> - "Users may be confused or misled by seeing actions linked to non-existent
>   governance domains, potentially causing incorrect data governance decisions." (Medium)
> - "Persistence of obsolete actions could clutter the interface and degrade user
>   experience, impacting productivity." (Low)

The first predicted risk is the genuine one. The second is a paraphrase / "soft
restatement" of the same problem from a different angle — a common LLM pattern
when asked for an *array* of risks: it pads to 2 even when 1 is correct.

**Hypothesis.** Schema asks for `"risks": array` with no minimum/maximum and the
prompt uses examples that imply multiple risks per item. The model treats "give
me risks" as "give me ≥ 2 risks".

**Mitigation candidates** (not yet attempted):
- Tighten prompt: "Return the smallest set of *distinct, non-overlapping* risks.
  Many work items have exactly one risk; do not pad."
- Add a few-shot example with a single-risk item.
- Post-process: dedupe risks by semantic similarity before scoring (a kinder
  scorer, not a real fix).

## 2. Severity over-predicted toward "High"

**Symptom.** Predicted severity distribution skews high relative to labels:

| Severity | Expected (labels) | Predicted |
|---|---:|---:|
| High | 1 | 9 |
| Medium | 16 | 30 |
| Low | 8 | 3 |

On the 12 risks where predictions matched, severity was correct only **41.7 %**
of the time, with the most common error being Medium → High.

**Hypothesis.** The prompt's High-severity definition ("blocking issues, security
concerns, data loss potential, deadline at risk") sounds applicable to many bugs
in the abstract, so the model anchors high. It also doesn't see the actual ADO
`Severity` field (`2 - High`, `3 - Medium`, `4 - Low`) — the labeler does.

**Mitigation candidates:**
- Pass the ADO `Microsoft.VSTS.Common.Severity` value into the prompt as a hint.
- Re-tune the severity definitions: emphasize that customer-facing UI bugs
  without data-loss risk are usually Medium or Low.
- Calibration few-shots showing realistic Low-severity examples.

## 3. Action items: 2× too many, semantic-but-not-keyword match

**Symptom.** Action F1 = 0.11 (P 0.08, R 0.18). Model produces **3.44 action
items / item**, labels expect 1.56. But **most predicted actions are sensible** —
the metric is keyword-strict.

**Example** — work item 3921445:

> **Expected action keyword groups** (must match all keywords in any one group):
> - `["remove", "actions", "deleted governance domain"]`
> - `["fix", "actions", "display", "governance domain"]`
>
> **Predicted action_items**:
> - "Investigate the backend process that handles deletion of governance domains and associated actions."
> - "Implement a fix to ensure actions linked to deleted governance domains are removed from the Actions tab automatically."
> - "Test the fix thoroughly to confirm that no actions remain visible after domain deletion."
> - "Deploy the fix to production and monitor for recurrence of the issue."

The 2nd predicted action contains "fix", "actions", "removed/governance domain"
— a human would call that a match. The matcher requires substring on each
keyword in the same item, so plurals/inflections ("removed" vs "remove") and
phrasing ("Actions tab" vs "actions") cause misses.

**Hypothesis (split):**
- ~50 % of the gap is real: model invents generic boilerplate
  ("Test the fix thoroughly", "Deploy the fix to production") that doesn't
  appear in the source.
- ~50 % is metric strictness: keyword groups are unforgiving on inflection.

**Mitigation candidates:**
- Prompt: "Generate at most 2-3 action items, each tied to a specific code path,
  team, or artifact mentioned in the work item. Do not include generic steps
  like 'test', 'deploy', or 'monitor'."
- Scorer: lemmatize keywords before substring match (cheap), or use
  embedding-cosine match with a threshold (expensive but truer).

## 4. Effort: defaults to "Medium" when "Small" is correct

**Symptom.** Effort accuracy 60 % (15/25). Predicted distribution:
12 Small / 13 Medium. Expected: 22 Small / 3 Medium / 0 Large. The model
under-uses Small.

**Hypothesis.** The prompt's effort buckets ("Small < 1 day, Medium 1-3 days,
Large > 3 days") give the model no anchor for what these closed bugs actually
cost; it averages toward the middle. The labeler had access to the same fields
but came in lower because most of these items closed quickly with a small fix.

**Mitigation candidates:**
- Add the work item's resolution time (close date − create date) to the prompt
  as a strong signal.
- Few-shot with at least one short bugfix labeled Small.

---

## 5. Summary keyword coverage: model is more verbose than the labels

**Symptom.** Summary keyword coverage 67.9 %. Forbidden-claim violations: 0
(model never invents a feature/state that wasn't in the work item — good).

The misses cluster on items where the labeler's keywords were drawn from the
title (e.g., "pdgbugbash") but the model's summary used a paraphrased, more
human-readable version. This is a label/keyword problem more than a model
problem; the summaries themselves read fine on inspection.

**Mitigation candidate (label side):** strip noise tokens like `pdgbugbash:` /
`[Bug]` from titles before deriving summary keywords.

## 6. Infrastructure: deployment throttling masquerades as model latency

**Symptom.** Per-call latency on the eval was: 4 s, 2 s, 69 s, 2 s, then **21 of
the next 22 calls landed in 58–61 s**, for a total wall-clock of ~22 min on 25
items. That looked like a slow model but isn't.

**Diagnosis.** HTTP-level logging (see `tools/diagnose_latency.py`) showed the
deployment is provisioned at:

| header | value |
|---|---|
| `x-ratelimit-limit-requests` | **1 / 60 s** |
| `x-ratelimit-limit-tokens` | **1000 / 60 s** |

Every 3rd request returned `429 Too Many Requests` with `Retry-After: ~58s`.
The OpenAI SDK silently respects `Retry-After` and retries — making each "slow"
call actually a `fail → sleep ~60s → retry → success` pattern. The eval code
itself is fine; it's the deployment SKU.

**Implication.**
- Reported latency numbers in `evals/runs/.../report.md` are *pessimistic by ~30×*
  for any deployment with normal quota. The cost number ($0.083) is unaffected.
- Reproducing the eval on a small / free deployment will take ~22 min for 25
  items no matter what the code does.

**Mitigation.**
- **Real fix:** raise the deployment's TPM/RPM in the Azure portal
  (Foundry → `my-foundry-learn` → Deployments → `gpt-4.1-mini` → Edit).
  Free for pay-as-you-go up to the resource-level quota.
- **Code-side workaround:** none meaningful. We could add an in-process limiter
  to make retries explicit instead of silent, but it doesn't make the run faster.

## What's *not* a known failure (yet)

- **Forbidden-claim violations:** 0. The model is not inventing features.
- **No infinite loops, no truncated JSON, no auth flakes** observed.
